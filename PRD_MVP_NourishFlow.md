# NourishFlow MVP · PRD

> 不劝退、可追溯、能陪伴——一个能持续认识你、监督饮食、给出有依据建议的营养助手。

- 文档版本：MVP v1.0
- 撰写日期：2026-05-06
- 项目定位：**自用为主，验证核心闭环**
- 工期：以"做完做对"为准，不强行卡时间
- 替代版本：取代 PRD v1.0 的全部 V1 范围

---

## 一、产品价值观（不可变更）

1. **不提倡轻断食、不提倡快速减脂**
2. **身体健康是唯一终极目标**，体重数字不是
3. **尊重用户每一个选择**——奶茶、零食、外卖都给最优解，不劝退、不羞辱
4. **饮食是生活，不是任务**——零打卡、零连续天数、零排行榜
5. **能力有限就承认有限**——不追求完美,追求"对当下情境可行 + 安全"
6. **数据透明**——每条建议可追溯到知识来源
7. **长期陪伴优先于短期效果**——核心价值是持续认识用户,不是一两次推荐做得多好

---

## 二、MVP 要回答的核心问题

这个 MVP 不是给团队 demo 的展品,是给你自己用一段时间后,判断这个产品方向值不值得继续做的实验。它需要回答:

1. **核心价值能不能立起来**:单 LLM + 全文检索 + 引用校验,质量够不够"能用"?
2. **不劝退人设能不能在 prompt 层面稳住**:不需要多 Agent 投票,纯 prompt + 程序化护栏够不够?
3. **长期陪伴感能不能做出来**:跨会话事实记忆 + 饮食历史 + 目标追踪,能不能让 AI 真的"认识你"?
4. **架构留白是否到位**:后续想升级到 RAG / 加渠道 / 加更精细人设时,能不能不重写?

## 三、MVP 核心场景(3 个,不做更多)

### 场景 1:奶茶决策

> **用户**:"今天想喝奶茶"
> **期望**:得到一个具体推荐方案(甜度、时段、推荐量),带 WHO / 膳食指南的引用,语气不教训。如果用户最近喝过奶茶/吃过高糖食物,AI 能感知到。

### 场景 2:外卖点餐

> **用户**:"想点麦当劳,吃啥比较不亏"
> **期望**:在用户已选定品牌的前提下,给出餐组合建议,结合用户画像(BMI、过敏)、目标(如减脂)、最近饮食记录,带文献引用。

### 场景 3:零食选择

> **用户**:"下午饿了,想吃点零食"
> **期望**:给替代选项(坚果 vs 饼干 vs 水果等),结合时段、用户画像、最近饮食,带引用。

**砍掉的场景**:食堂(菜品没标准)、自己做饭(食谱推理复杂)。这两个 V1 再加。

## 四、MVP 主动放弃的事

下面这些事 MVP 阶段**不解决,并接受这个缺陷**:

- ❌ **不做角色立绘 / 软萌人设语气**——MVP 阶段就是普通 AI 助手语气,等核心稳了再加人设 prompt
- ❌ **不做多渠道**——只在网页上能用,推迟到 V1
- ❌ **不做用户系统**——本地单用户,不做注册登录,直接用 user_id
- ❌ **不做向量检索**——全文检索 (Meilisearch) 够了,V1 再加
- ❌ **不做多 Agent / 投票 / 校验**——单 LLM + 程序化护栏
- ❌ **不做 query 改写**——用户原话直接搜
- ❌ **不做主动饮食记录(确认气泡那一套)**——只做被动版,从对话里提取
- ❌ **不做会话历史浏览 UI**——数据存了,但前端不展示,V1 再加
- ❌ **不做卡路里 / 营养素精确计算**——文献依据为主,数字估算为辅
- ❌ **不做对话态价值观智能切换**——靠 prompt 自然处理,极端 case 可能翻车
- ❌ **不做查询日志分析、监控告警、成本追踪**——MVP 阶段失败就失败,自己看日志
- ❌ **不做缓存层**——每次对话都重新检索 + LLM 调用
- ❌ **不做引用幻觉重试**——校验失败只记录
- ❌ **不做 PubMed 爬虫**——MVP 只手动上传 8-10 篇 T1 文献
- ❌ **不做"待确认区"画像确认机制**——AI 提取直接入库,用户在画像页可改可删
- ❌ **不做 confidence / evidence_count 详细评分系统**——简化为最基础的字段更新

## 五、MVP 功能模块

### 5.1 模块 1:用户画像(静态 + 动态)

#### 5.1.1 静态画像(注册时填)

5 个字段,首次进入必填:
- 身高 cm
- 体重 kg
- 出生年
- 性别(男/女/其他)
- 过敏原(多选 + 自填)

提供画像设置页,可随时修改。

#### 5.1.2 动态画像(对话中自动学习)

**核心原则**:AI 从对话中持续学习用户的事实,跨会话保留,这是"长期陪伴"的根本。

**4 类提取范围**:

| 类别 | 例子 | 提取方式 | 入库阈值 |
|------|------|---------|---------|
| **关键安全(critical)** | 过敏、慢病、用药、孕期 | **同步阻塞** | 1 次明确表达 |
| **目标(goal)** | 想减脂、改善血糖、增肌、控糖 | **同步阻塞** | 1 次明确表达 |
| **强烈偏好(preference)** | 素食、清真、不吃内脏、讨厌香菜 | 异步 | 1 次明确表达 |
| **生活习惯(habit)** | 经常熬夜、久坐工作、外卖多 | 异步 | 累积 2 次 |

**为什么 critical 和 goal 必须同步**:
- 下一句话就可能问"推荐零食",画像里没过敏会出大事
- 目标(如"想减脂")直接影响整个对话的推荐方向,不能延迟生效

**为什么 preference 和 habit 异步**:
- 不影响安全,慢慢学
- 异步用小模型(DeepSeek-V4),不阻塞主对话延迟
- habit 类需要累积 2 次避免误判("今天有点累" ≠ "经常累")

**关键设计**:
- 已存在的字段被新信号覆盖时,以最新的为准(last_updated_at)
- 用户在画像页编辑过的字段,提高更新阈值(critical 仍同步,其他需要 3 次累积才覆盖)
- 用户在画像页删除的字段,软删除,AI 不再使用,但记录保留防误删
- **不做**"待确认区"、不做 confidence 评分、不做复杂的"永不覆盖"逻辑

**画像注入 LLM 的策略**:
- critical 字段(过敏、慢病、用药、孕期)**永远注入**到 system prompt
- goal 字段**永远注入**(影响整个对话方向)
- preference 字段按相关性注入(用户问奶茶时注入"咖啡因敏感"等)
- habit 字段按相关性注入(用户问晚餐时注入"经常熬夜")
- 不注入的字段在 prompt 中省略,节省 token

#### 5.1.3 画像设置页

提供基础 CRUD:
- 5 个静态字段:可编辑
- 动态归纳的所有字段:可看见、可编辑、可删除
- 删除后该字段不再用于推荐

**MVP 不做**:
- ❌ 三段式分区显示(静态/我告诉过你的/AI 了解的我)
- ❌ 置信度星级显示
- ❌ "✓ 准确 / ✗ 不对了" 三选项
- ❌ evidence 原话回显

V1 再加这些"可解释面板"的精细 UI。

### 5.2 模块 2:对话与记忆

#### 5.2.1 对话历史(L1:本轮上下文)

- 后端管理会话边界,不依赖前端拼接
- 每条消息入库 messages 表,关联 conversation_id
- 用户主动点"新对话"按钮 → 创建新 conversation_id
- 主对话调用时取**当前会话最近 10 轮**注入 prompt
- 超过 token 预算时直接截断最早的(MVP 不做摘要压缩)

**MVP 不做**:
- ❌ 30 分钟无活动自动新会话(V1 加)
- ❌ 跨会话语义关联 / 长会话摘要压缩(V1 加)

#### 5.2.2 跨会话事实记忆(L2)

见 5.1.2 动态画像。

#### 5.2.3 饮食历史记忆(L3:被动版)

**核心原则**:不主动问、不弹确认、不强迫记录。但用户在对话里自然提到吃了什么时,AI 后台异步提取并入库。

**触发逻辑**:
- 关键词扫描("吃了"、"刚吃"、"点了"、"在吃"、"晚餐"、"午饭"等)
- 命中后异步用小模型提取:时间、餐别、食物列表
- 直接入库 meal_logs 表,不弹确认气泡
- **不做摄入估算**(卡路里、营养素),只做事实记录

**注入主对话**:
- 每次主对话加载用户**最近 3 天的饮食记录**注入 prompt
- 让 AI 自然带入上下文:"你昨天午餐已经吃了麻辣烫,今晚清淡点"

**MVP 不做**:
- ❌ 主动问"今天吃了什么"
- ❌ "刚才吃了对吧"确认气泡
- ❌ 卡路里 / 营养素估算
- ❌ 周报 / 月报
- ❌ 健康曲线可视化
- ❌ 饮食历史浏览页

V1 视情况加主动版。

#### 5.2.4 不做的层级(L4:情感/互动记忆)

MVP 阶段 AI 不感知用户最近压力大,不感知互动满意度。这部分推迟到 V2。

### 5.3 模块 3:知识库

#### 5.3.1 起步内容(8-10 篇 T1 文献)

T1 必备:
- 中国居民膳食指南 2022 完整版
- WHO Healthy Diet Fact Sheet
- WHO Sugars Intake Guideline 2015
- Dietary Guidelines for Americans 2020-2025

T2 重要:
- AHA 心血管饮食推荐
- 中国营养学会几个共识文件(2-3 篇)

如果 MVP 跑起来发现召回不足,针对性补 1-2 篇主题文献(含糖饮料、加工食品等)。

#### 5.3.2 入库流程

```
PDF 文件
   ↓
解析(pdfplumber)+ 抽元数据(LLM 一次)+ 分块
   ↓
写入 PostgreSQL(articles + article_chunks)
   ↓
同步索引到 Meilisearch(chunk_id + content + 部分元数据)
   ↓
返回入库成功
```

**分块策略**:
- 固定大小:800 字 + 100 字 overlap
- **不做** semantic chunking
- **不做** chunk_type 分类

**元数据**:
- 必填:tier(1-4)、source_org、title
- LLM 自动抽:pub_year、authors
- 入库时记录:section_title(如能从 PDF 抽到)、page_number

**入库工具**:命令行脚本(独立开发者一次性运维),无需做后台管理 UI。

#### 5.3.3 数据一致性

- 先写 PG 后同步 Meilisearch
- 同步失败重试 3 次,仍失败写入"待同步"队列 + 日志
- Meilisearch 挂了不丢数据,重启后从 PG 重建索引
- **不做**强一致(滞后几秒可接受)
- **不做**消息队列(MVP 单机异步任务即可)

### 5.4 模块 4:检索 + 主对话流程

#### 5.4.1 完整流程

```
用户消息进入
    │
    ├─→ [关键词扫描] 同步,毫秒级
    │     └─→ 命中 critical/goal → [同步 LLM 提取 + 入库画像]
    │
    ├─→ [主对话流程] 同步
    │     ├─→ 加载画像(含动态归纳)
    │     ├─→ 加载会话历史(最近 10 轮)
    │     ├─→ 加载最近 3 天饮食记录
    │     ├─→ 全文检索 chunks(Meilisearch)
    │     ├─→ Tier 加权重排
    │     ├─→ 组装 system prompt → LLM 流式调用
    │     └─→ 流式返回 + 入库 messages
    │
    └─→ [异步后处理] 不阻塞返回
          ├─→ 提取一般偏好/习惯 → 累积阈值入库
          ├─→ 提取饮食记录 → 入库 meal_logs
          └─→ 引用校验 → 失败仅记录日志
```

#### 5.4.2 检索

- 用 Meilisearch 全文检索,用户原话直接搜
- 过滤:tier 范围、可选 source_org
- top-5 召回 + tier 加权重排(程序化)
- 返回 chunks 给 LLM,带元数据

**MVP 不做**:
- ❌ Query 改写
- ❌ 多 query 合并
- ❌ 向量检索(V1 加 pgvector)
- ❌ BM25 + 向量混合
- ❌ 重排模型

#### 5.4.3 主 LLM 调用

- 模型:**Claude Sonnet 4.5**(主推荐)
- 备选:DeepSeek-V4(成本敏感时切换)
- 流式输出
- system prompt 组成:
  - 价值观与硬底线
  - 用户画像(critical 永远注入,其他按相关性)
  - 最近 3 天饮食记录
  - 检索到的 chunks(每个标 chunk_id)
  - 输出格式要求(必须带 [chunk_id:xxx] 引用)

#### 5.4.4 引用机制

- LLM 输出强制要求 `[chunk_id:xxx]` 格式
- 后端正则提取 cited_chunk_ids
- 校验 chunk_id 是否真实存在 + 是否在本次检索结果里
- 校验失败:**仅记录日志,不重试不阻断**(MVP 接受偶发幻觉,V1 再加重试)

#### 5.4.5 程序化安全护栏

- 检测推荐里的食物 → 食物-成分映射表查 → 与画像过敏列表比对 → 命中即拒绝并要求 LLM 重生成
- 卡路里硬底线:< 1200 kcal/日方案直接拒绝
- "快速减肥"等极端关键词:静默规避,不响应
- 自伤/极端关键词:中止饮食功能 + 显示援助资源

**MVP 简化版**:食物-成分映射表手工维护一个最小集(几十种主要食物的过敏原 / 重要成分),覆盖你画像里填的过敏类型即可。V1 再扩展。

- 一个聊天页(消息列表 + 输入框 + "新对话"按钮)

## 六、MVP 架构

### 6.1 技术选型

| 层 | 选型 | 备注 |
|---|---|---|
| 后端 | Python 3.11 + FastAPI | 异步原生 |
| 主存储 | PostgreSQL 15 | 数据真理之源 |
| 搜索索引 | **Meilisearch** | 中文友好、typo tolerance |
| LLM 调用 | LiteLLM | 多 provider 统一接口 |
| LLM 主模型 | Claude Sonnet 4.5 | 主对话生成 |
| LLM 提取模型 | DeepSeek-V4 | 异步提取、元数据抽取 |
| 前端 | React + Vite + Tailwind + shadcn/ui | 默认风格 |
| 部署 | Docker Compose(单机) | PG + Meilisearch + 后端 + 前端 |

**故意不引入**:
- Redis(无缓存需求)
- Qdrant / pgvector(V1 才上向量)
- LangGraph / nanobot(无 Agent 编排需求)
- 消息队列(无异步任务复杂度)
- 自研网关层(LiteLLM 够用)

### 6.3 数据存储分工

**PostgreSQL**(7 张表):
- `users` — 单用户即可,先不做认证
- `user_profile_static` — 5 个静态字段
- `user_profile_dynamic` — 动态归纳的字段
- `articles` — 文献元数据
- `article_chunks` — 切块内容(预留 embedding 字段,V1 用)
- `conversations` — 会话边界
- `messages` — 对话记录(含引用 chunk_ids)
- `meal_logs` — 饮食被动记录

**Meilisearch**(1 个索引):
- `chunks` 索引
- 字段:chunk_id, content, article_title, source_org, tier, pub_year, section_title
- 可搜索:content, article_title, section_title
- 可过滤:tier, source_org, pub_year

### 6.4 关键架构决策

**1. PG 是主存储,Meilisearch 是索引**
- 任何数据写入先 PG 后 Meilisearch
- Meilisearch 挂了不丢数据,重建即可
- 检索时 Meilisearch 返回 chunk_id,回 PG 拉完整内容

**2. 检索能力封装为统一接口**
- 业务层不直接调 Meilisearch
- V1 加向量检索时,接口不变,内部实现切换或混合

**3. 引用机制依赖 LLM 输出格式**
- system prompt 强制要求 `[chunk_id:uuid]` 格式
- 后端正则提取 + PG 校验
- LLM 不遵守格式时输出会失去引用——MVP 接受这个偶发问题

**4. 不做异步任务编排**
- 异步提取用 `asyncio.create_task` 起后台任务
- 不引入 Celery / RQ
- 单机部署,单 worker 进程够用

**5. 关键安全检查不靠 LLM**
- 过敏检测用食物-成分映射表 + 程序化匹配
- 卡路里底线用硬规则
- 极端目标用关键词列表
- LLM 只做内容生成,不做安全判断

### 6.5 V1 升级路径(架构留白)

| V1 升级方向 | MVP 留的位 | 升级成本 |
|---|---|---|
| 加向量检索 | chunks 表预留 embedding 字段;检索接口可扩展 | 中(需跑全量 embedding) |
| 加 query 改写 | 检索接口前加一步即可 | 低 |
| 加角色人设语气 | system prompt 可改 | 极低(改 prompt) |
| 加多渠道(飞书等) | 后端是纯 API | 中(加 nanobot 或 webhook) |
| 加移动端 PWA | React 项目可加配置 | 低 |
| 加可解释画像面板 | 动态画像数据已存 | 中(前端加 UI) |
| 加会话历史浏览 | messages/conversations 数据已存 | 低(前端加 UI) |
| 加主动饮食记录 | meal_logs 表已存 | 低(加确认气泡) |
| 加周报/月报 | meal_logs 数据已存 | 中 |

## 八、安全 / 合规(MVP 自用阶段)

| 项 | MVP 自用 |
|---|---|
| 极端方案拦截 | 硬规则:< 1200 kcal/日拒绝 |
| 极端目标拦截 | 关键词列表:"快速减肥"、"一周瘦 X 斤"等静默规避 |
| 心理危机识别 | 关键词列表(自伤/极端)→ 中止饮食功能 + 显示援助资源 |
| 过敏冲突拦截 | 食物-成分映射表 + 程序化匹配 |
| 知识库审核 | 你自己上传,你自己信 |
| 数据加密 | 应用层不加密 |
| 用户数据导出/删除 | MVP 不做(自用) |
| 医疗免责 | UI 上一行小字 |

**自用阶段的硬底线(即使自用也不破)**:
- ❌ 不出 < 1200 kcal/日方案
- ❌ 不响应"快速减肥"目标
- ❌ 检测到自伤/极端关键词 → 中止饮食功能 + 显示援助资源
- ❌ 命中过敏成分 → 拒绝并重生成

## 九、MVP 验收标准

### 9.1 功能验收(二元判断)

- [ ] **入库**:能成功上传 8-10 篇 PDF 到 PG + Meilisearch,元数据正确
- [ ] **静态画像**:能填 5 字段、能修改、能删除
- [ ] **动态画像**:在对话里说"我对花生过敏",新会话画像已自动记录
- [ ] **目标**:在对话里说"我想减脂",新会话推荐方向已对齐
- [ ] **饮食被动记录**:在对话里说"中午吃了麻辣烫",下次对话 AI 知道这件事
- [ ] **多轮对话**:"想喝奶茶" → "加珍珠呢" 能正确理解上下文
- [ ] **检索**:3 个核心场景的 query 都能从 Meilisearch 召回相关 chunks
- [ ] **对话**:3 个核心场景每个都能拿到流式回答 + 引用徽章
- [ ] **引用**:徽章可点击查看原文片段、来源、tier
- [ ] **校验**:LLM 输出的 chunk_id 在 PG 中真实存在
- [ ] **过敏拦截**:画像里有"花生过敏"后,推荐不会出现花生制品
- [ ] **极端目标拦截**:用户说"想一周瘦 10 斤",AI 不会响应该目标

### 9.2 质量验收(自用一周后判断)

| 指标 | 目标 | 不达标说明 |
|------|------|------------|
| **推荐接受率** | ≥ 60% | 你自己使用一周,60% 的推荐你愿意采纳 |
| **引用真实率** | ≥ 95% | 校验失败 / 幻觉 chunk_id 比例 < 5% |
| **价值观稳定性** | 0 次失控 | 不会突然出现劝退、教训、羞辱 |
| **动态归纳准确率** | ≥ 90% | 对话里明确表达的事实,AI 能正确入库 |
| **错误归纳率** | ≤ 5% | 不该入库的事被入库的概率 |
| **响应时间 P50** | ≤ 5 秒 | 用户输入到第一个 token 返回 |
| **响应时间 P95** | ≤ 10 秒 | 完整回答返回 |

**软性指标(自用感受)**:
- 用了一周后,你**会主动想用它**做饮食决策(而不是觉得麻烦)
- 推荐的内容**让你学到了**新东西
- 引用机制**让你信任**它的建议
- AI **真的认识你**——会用你之前说过的事实、追踪你的目标、知道你最近吃了什么

### 9.3 失败标准(任一触发,停下重评估)

- ❌ 跑通 3 个核心场景超过预期工期 50%
- ❌ 推荐接受率 < 40%
- ❌ 引用幻觉率 > 15%
- ❌ 出现 1 次以上严重价值观漂移(教训用户、推荐危险方案)
- ❌ 动态归纳错误率 > 15%
- ❌ 响应时间 P95 > 20 秒

任何一条触发,**停下来重新评估,不要硬推 V1**。

## 十、风险与对应

| 风险 | 应对 |
|------|------|
| LLM 引用幻觉 | 后端校验 chunk_id 真实存在 + 是否在召回结果里;MVP 仅记录不重试 |
| 价值观失控 | system prompt 写明硬底线;定期人工抽查;程序化护栏 |
| 全文检索召回不足 | 监控查询日志(V1 加),针对性补充文献;预留向量检索升级路径 |
| 8 篇文献覆盖不够 | MVP 跑通后按需补 2-3 篇主题文献 |
| 单 LLM 调用质量不稳 | MVP 接受偶发不稳;V1 再考虑校验机制 |
| 异步动态画像漏提取 critical | 关键词扫描兜底,critical/goal 走同步阻塞 |
| 异步提取误判("今天累" → "经常累") | habit 类需累积 2 次;preference 1 次但易于用户在画像页删除 |
| Meilisearch 与 PG 不一致 | 入库后立即同步 + 失败重试 3 次;V1 加每周对账 |
| 食物-成分映射表覆盖不全 | MVP 只覆盖你画像里填的过敏类型;V1 系统化扩展 |
| 工期延迟 | 严格不加功能;到点不论完成度先停下评估 |

## 十一、MVP 之后

MVP 跑通 + 自用一周后,按价值密度排序加(每一波 1-2 周):

1. **可解释画像面板**——三段式 UI、置信度、evidence 显示、待确认区
2. **会话历史浏览**——能回看之前的对话
3. **主动饮食记录**——确认气泡、卡路里估算、营养素分析
4. **角色人设语气**——软萌跟班 prompt + 简单立绘 / emoji
5. **向量检索混合**——pgvector + 全文混合检索
6. **多渠道接入**——飞书 / 微信公众号 / PWA
7. **Query 改写 + 重排器**——RAG 升级
8. **周报 / 月报 / 健康曲线**——长期数据可视化
9. **食物-成分映射表完整化**——支持更细粒度的安全检查
10. **PubMed 爬虫**——自动扩充知识库

每一波之后都重新评估要不要继续。
