## 1. 四条核心原则

### 1.1 想清楚再写(Think Before Coding)

- 不要假设。不要藏起困惑。把权衡说出来。
- 有多种解读时,**列出来,不要默默选一个**。
- 有更简单的做法时,说出来,适当反驳我。
- 不清楚就停下来,说清楚哪里不清楚,然后问。

### 1.2 简单优先(Simplicity First)

- 只写要求做的功能。**没要求的功能不写**。
- 不为单次使用的代码做抽象。
- 不加我没要求的"灵活性"或"配置项"。
- 不为不可能发生的场景做错误处理。
- 写了 200 行如果能 50 行解决,**重写**。

自检:"一个资深工程师会说这过度设计吗?" 会就简化。

### 1.3 外科手术式修改(Surgical Changes)

**只改必须改的。只清理你自己造的烂摊子。**

- 不"顺手优化"邻近代码、注释、格式。
- 不重构没坏的东西。
- 匹配现有代码风格,即使你觉得自己的更好。
- 发现无关的死代码,**告诉我,不要直接删**。
- 你的改动让某些 import / 变量 / 函数变成孤儿,**只清理你造成的孤儿**,不要清理项目里已有的死代码。

测试:每一行变更都能直接追溯到我的需求吗?

### 1.4 目标驱动执行(Goal-Driven Execution)

把任务转成**可验证的目标**:

- "加校验" → "为非法输入写测试,然后让它通过"
- "修这个 bug" → "写一个能复现 bug 的测试,然后让它通过"
- "重构 X" → "前后跑同一个测试,都通过"

多步任务先列计划:
```
1. [步骤] → 验证:[检查项]
2. [步骤] → 验证:[检查项]
```

强成功标准 → 你能独立闭环。弱标准("能用就行")→ 你会反复来问我,我烦。

## 2. 工作循环(Phase 1 / 2 / 3)

我们按开发 PRD 第六节的 5 个 Phase(P1~P5)推进。**每个 Phase 走一遍完整循环**:

### Phase A · 动手前对齐【必做,不能跳】

- 复述这个 Phase 的目标、产出、验收标准(从开发 PRD 抄)
- 列出你要新建/修改的文件清单
- 指出 PRD 没说清楚的歧义,**问我**
- **等我说"开始"再写代码**

### Phase B · 编码

- 严格按开发 PRD 的目录结构和契约写
- **不要顺手优化** PRD 没要求的东西(加缓存、加重试、加日志中间件、改架构……)
- 遇到 PRD 没覆盖的决策,**停下来问**,不要自己拍

### Phase C · 自审【硬清单,不写感想】

按下面清单逐条 ✅ / ❌,**不允许"基本满足"**:

- [ ] 接口签名和开发 PRD 一致
- [ ] 没有 PRD 之外的新文件
- [ ] 没有 PRD 之外的新依赖
- [ ] 错误路径有处理(不是 `try/except: pass`)
- [ ] 关键留位字段保留(`chunk_type`、`cited_chunk_ids`、`embedding` 占位等)
- [ ] MinerU 接缝完整(`parse_pdf` 函数签名未变)
- [ ] 没改 PRD 标记为"砍掉"的功能(画像/饮食/三数字卡/护栏等)

任何一条 ❌,**先修,再继续**。

### Phase D · 验收【按 PRD 第六节的"验收"列跑】

- 跑真实端到端验收命令(curl / CLI / 浏览器手测)
- **不要现编"假装通过"的单元测试**来糊弄
- 跑不过就贴**真实报错**,不要说"应该可以"

### Phase E · 报告

```markdown
### Phase X 测试报告

**验收项**
- ✅ /api/health 返 200
- ✅ ingest CLI 跑通,PG 有 5 条 chunks
- ❌ Meili 搜不到中文 —— 原因:索引未配 zh tokenizer

**实际命令和输出**
$ curl ... 
{ ... }

**偏离 PRD 的地方**
(无 / 列出来说明原因)

**下一步建议**
继续 P{X+1} / 修复某项 / 找你讨论 XX
```

### Phase F · 架构对齐【只在岔路口】

**不是每个 Phase 都讨论架构**。只在以下情况停下来找我聊:

- 发现开发 PRD 的设计在实现时跑不通
- 有 ≥2 种合理实现,选哪个会影响后面 Phase
- 性能或正确性有隐患
- 用户行为或外部依赖比 PRD 假设复杂

平时跑顺了直接进下一个 Phase,**不用反复确认**。

## 3. 项目特定红线

### 3.1 不许擅自做的事

- ❌ 加 PRD 没列的依赖(尤其是 redis / celery / qdrant / 任何向量库)
- ❌ 加 PRD 没列的表(画像表、饮食表、血糖表全砍了)
- ❌ 加"安全护栏"的关键词扫描逻辑(MVP 在 prompt 里写一句兜底就够)
- ❌ 加引用真实性校验的 LLM 二次调用
- ❌ 把 FastAPI 容器化加进 docker-compose(今天本地 `uv run` 起)
- ❌ 用 alembic(今天 `SQLModel.metadata.create_all` 就行)
- ❌ 给 LLM 加 fallback 多模型路由(今天就 DeepSeek 一个)
- ❌ 引入复杂的 SSE 框架(`sse-starlette` 够了)

### 3.2 必须保留的接缝

这些是给 V1 留的口子,**今天不能为了"简洁"砍掉**:

- `parse_pdf(path) -> list[Section]` 函数签名 —— 明天换 MinerU 的唯一接缝
- `article_chunks.chunk_type` 字段 —— 今天全是 `text`,明天加 figure/table
- `article_chunks.embedding` 留位(VECTOR 类型) —— 今天不写,明天向量检索用
- `messages.cited_chunk_ids` JSONB —— 今天就要写入

### 3.3 提示词修改流程

如果你要改 system prompt(`app/services/chat.py` 里的那段),**写成 diff 给我看,我同意才改**。提示词是产品逻辑,不是实现细节。

## 4. 沟通规范

### 4.1 你给我的输出

- 不要长篇 markdown 报告夹杂一堆 emoji。简洁、直接。
- 代码 diff 只展示**关键部分**,完整代码我自己看文件。
- 不确定的地方明确说"不确定",不要硬撑。
- 跑了命令贴**真实输出**,不要复述"成功了"。

### 4.2 你不要做的事

- 不要在我没问的时候主动给"下一步建议"。
- 不要在每条消息末尾写"还有什么我可以帮你?"
- 不要为了显得有用,虚构进度("我已经验证过了")—— 没跑就说没跑。
- 不要用"应该""可能""理论上"来包装你没验证的结论。

## 5. 出错时怎么办

### 5.1 同一个 bug 修了 3 次还在

**停下来**。这是信号:你对系统的理解是错的。
告诉我现状,一起重新梳理,不要继续盲改。

### 5.2 跑不通某个外部依赖(Meili / DeepSeek / pdfplumber)

- 先确认是配置问题还是代码问题
- 贴真实报错和对应的环境变量
- 不要直接换库("Meili 不行换 ES 吧") —— PRD 选型已定,问我再说

### 5.3 PRD 漏了一个边界条件

- 在你的报告里明确写"PRD 没覆盖 X 情况,我按 Y 处理了"
- 不要默默处理后不说
